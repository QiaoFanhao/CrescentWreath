# AGENTS.md

本文件用于指导 Codex / 代码代理在本仓库中的工作方式。目标不是“尽快堆功能”，而是**严格按设计文档推进《新月花冠》电子版 Unity 项目**，避免再次出现“想到什么做什么、系统边界混乱、规则散落硬编码”的问题。

---

## 1. 项目目标

本项目是《新月花冠》电子版，当前目标是：

- 平台：Unity
- 类型：4人固定 / 2V2 / 联机对战卡牌游戏
- 核心原则：**服务器权威、数据驱动、规则显式建模、按详细设计文档开发**
- 当前工作方式：**先搭骨架，再逐模块实现，再做联调**

Codex 在本项目中不是“自由发挥型生成器”，而是**严格执行设计与工程规范的协作开发者**。

#### 核心规则

若命名不确定，宁可先查仓库和文档，不要自行创造。

## 4. 开发原则：先骨架，后实现；先显式模型，后特例

### 4.1 优先做骨架
当某个系统还未落地时，优先实现：
- 数据结构
- 枚举
- 状态对象
- 接口边界
- 流程占位
- 日志与调试入口

不要一上来就把某张牌、某个角色的完整效果硬塞进去。

### 4.2 优先做通用规则，不要先写个例硬编码
比如：
- 先做 `DamageContext`，不要先只写某一张伤害牌
- 先做 `ResponseWindow`，不要先只写某角色响应技
- 先做 `OverlayContainer`，不要只写机械翡翠一张牌的特殊脚本

### 4.3 特例必须挂在通用框架上
例如：
- 《世纪末风神传说》应挂在“非 `Barrier` 的伤害免疫替代效果”时点上
- `Seal` 应作为防御值运行时修正器，而不是某处临时 `-1`
- `Shackle` 应在回合开始流程中创建 `InputContext`，而不是直接在某个 UI 按钮里特判

## 5. 代码架构要求

### 5.1 服务器权威优先
凡涉及规则裁定的逻辑，优先放在**规则层 / 状态层 / 服务器逻辑层**，而不是 UI 层。

客户端职责应主要包括：
- 展示 `VisibilityView`
- 收集输入
- 发送请求
- 展示日志 / 动画 / 提示

客户端不应：
- 自己推导最终规则结果
- 自己决定是否合法
- 自己偷偷修改权威状态

### 5.2 规则逻辑与表现逻辑分离
必须尽量保持：
- 规则数据结构与裁定逻辑
- Unity MonoBehaviour / 动画 / 交互表现

两者分离。

禁止把大量核心规则直接写在：
- UI Button 回调里
- View 层脚本里
- 临时测试 MonoBehaviour 里

### 5.3 状态对象必须可序列化 / 可恢复
下列对象应优先按“可保存、可重连恢复”的思路设计：
- `GameState`
- `ActionChain`
- `ResponseWindow`
- `InputContext`
- `DamageContext`
- 状态异常对象
- `currentAnomalyState`

### 5.4 优先数据驱动
能通过配置 / 定义层表达的内容，不要先写死在 if-else 链中。

尤其这些系统要优先朝数据驱动设计：
- 技能定义
- 宝具定义
- 异变定义
- 效果节点
- 状态定义

但也不要为了“纯配置”而把简单逻辑搞得不可读；原则是：
**结构化优先，必要时脚本扩展兜底。**

## 6. Codex 的具体工作方式

### 6.1 每次任务的默认输出结构
除非用户明确要求只给代码，否则默认按以下结构工作：

1. **先说明你准备改什么**
2. **列出会改哪些文件**
3. **再实际修改代码**
4. **最后总结这次改动完成了什么、还缺什么**

### 6.2 改动前先搜索
在动手前优先搜索：
- 是否已有同名 / 同职责结构
- 是否已有旧实现可复用
- 是否已有命名规范对应对象
- 是否已有测试 / 占位逻辑

不要重复造轮子。

### 6.3 改动应尽量小而完整
- 不要为了一个点顺手大面积重命名无关文件
- 但也不要只改一半，留下明显断裂
- 优先做“最小但闭合”的增量改动

### 6.4 若发现设计冲突，先停下并报告
如果发现：
- 文档之间冲突
- 旧代码与文档根本不兼容
- 当前任务会强迫你先决定未定义规则

则不要擅自拍板。应先输出：
- 冲突点
- 影响范围
- 建议的处理方向

## 7. 禁止事项（非常重要）

### 7.1 禁止“顺手优化”为另一套架构
不要因为你觉得某种模式更熟，就把本项目偷偷改写成：
- 你自己偏好的 ECS 架构
- 纯事件总线乱发消息架构
- 纯脚本驱动无状态对象架构
- 大一统单例游戏管理器神类架构

本项目的设计核心已经明确：
- 显式状态对象
- 显式规则链
- 联机权威裁定
- 可恢复中间态

### 7.2 禁止硬编码卡牌个例污染骨架
除非当前任务就是做某张牌的个例适配，否则不要把：
- 某张角色卡
- 某张宝具
- 某张异变

直接硬写进底层通用模块。

### 7.3 禁止静默修改规则
不要在不说明的情况下把规则改成“更好实现”的版本。比如：
- 把 `Barrier` 当普通防御
- 把 `Silence` 扩大成不能用所有技能
- 把 `Charm` 改成直到成功禁防一次才消耗
- 把《世纪末风神传说》并入标准 `Barrier`

规则一旦改了，必须明确说明。

### 7.4 禁止生成“看起来能跑”的假实现冒充完成
尤其禁止：
- 写空壳函数但声称逻辑完成
- 只在客户端本地模拟而声称系统完成
- 用 TODO 占位却在总结里说“已支持”

可以做骨架，但必须如实说明现在只是骨架。

## 9. 测试与验收标准

Codex 每次提交应至少满足以下“完成定义”：

### 9.1 编译通过
- 新增代码不应引入明显编译错误
- 若无法验证编译，必须明确说明“未实际编译验证”

### 9.2 命名一致
- 与命名规范文档一致
- 不引入新的混乱别名

### 9.3 规则链路闭合
例如做一个新系统时，至少要确认：
- 数据结构放哪
- 入口在哪
- 状态怎么更新
- 日志 / 事件怎么挂
- UI 或调用方怎么接

### 9.4 不破坏现有边界
- 不把规则塞进 UI
- 不把联机裁定塞进 View 层
- 不把局部特例写进无关底层

## 10. 当前开发优先级建议

若用户没有额外指定，默认按这个顺序推进：

### 第一优先级：地基可运行
- `GameState` 骨架
- 房间 / 对局状态
- 请求 / 消息模型
- `ActionChain` / `ResponseWindow` / `InputContext`
- 基本同步与重连恢复

### 第二优先级：核心规则可闭环
- 区域移动
- 伤害 / 防御 / 击杀
- 状态异常
- 角色技能入口
- 宝具 `Play / Summon / DefensePlace`
- 当前异变对象与解决流程

### 第三优先级：复杂规则与表现
- `OverlayContainer`
- `CharacterSetAside`
- 效果表达系统
- 日志与回放辅助
- 完整 UI / 交互表现

## 11. 当任务不明确时，Codex 应该怎么做

如果用户说“先做这个模块”，但没有说明具体做到什么程度，默认策略是：

1. 先补**最小可用骨架**
2. 再补**与现有文档对齐的关键接口**
3. 不要越过当前任务范围扩展到整套系统
4. 在总结里明确：
   - 已完成
   - 未完成
   - 下一步建议

## 12. 最后一句：本项目最重要的不是速度，而是结构不崩

以前已经踩过的坑是：
- 没有从设计文档开始
- 想到什么做什么
- 功能能跑但进度混乱
- 系统边界不清

所以从现在开始，Codex 在这个仓库里的第一职责不是“快”，而是：

> **让项目在不断增长时，仍然保持结构稳定、命名统一、规则显式、可继续开发。**

如果某个改动能更快做完，但会让系统边界变脏，那就不应该这样做。
