# 04_00_工程总览

## 1. 文档目的

本文档用于定义《新月花冠》电子版项目的整体工程结构、模块边界、依赖方向与开发顺序。

本文件不负责重新定义游戏规则，也不替代规则详细设计文档。
其目标是回答以下问题：

- 本项目由哪些工程组成
- 各工程分别负责什么、不负责什么
- 它们之间应如何依赖
- 当前阶段的实现顺序是什么
- 后续工程设计文档在整体中的位置是什么

本文档是 `04_工程设计` 目录的入口文档。
后续所有工程设计文档，均应以本文档所定义的总体原则为基础展开。

---

## 2. 项目背景

《新月花冠》电子版是一个基于 Unity 表现层、以服务器权威裁定为核心的联机卡牌游戏项目。

项目的基本定位如下：

- 平台：Unity
- 玩法：4人固定、2V2、联机对战
- 核心原则：
  - 服务器权威
  - 数据驱动
  - 规则显式建模
  - 中间态可恢复
  - 规则层与表现层分离
- 当前开发方式：
  - 先建立 `RuleCore`
  - 再接入 `Server`
  - 最后接入 `UnityClient`

本项目不是“先做一个能跑的客户端，再逐步补规则”的开发方式，
而是以规则核心为起点，逐步向网络与表现层扩展。

---

## 3. 总体工程拆分

项目整体拆分为以下几个主要部分：

```text
CrescentWreath
├─ docs
├─ RuleCore
├─ Server
└─ UnityClient
```

### 3.1 docs

文档目录。

负责保存：

* 规则书
* FAQ
* 详细设计文档
* 工程设计文档
* 后续开发规范与阶段性记录

`docs` 是项目的文字性来源，不参与运行时逻辑。

---

### 3.2 RuleCore

纯 C# 规则核心类库。

负责：

* 对局状态建模
* 动作请求建模
* 时序链与响应窗口建模
* 伤害 / 防御 / 击杀 / 状态等核心规则建模
* 规则结算与状态推进

不负责：

* UI
* Unity 场景与 MonoBehaviour 生命周期
* 网络连接
* 房间管理
* 动画与表现

`RuleCore` 是整个项目的规则真相核心，也是优先级最高的开发部分。

---

### 3.3 Server

服务器工程。

负责：

* 房间与对局会话管理
* 玩家连接管理
* 请求接收与分发
* 调用 `RuleCore` 进行权威裁定
* 维护对局权威状态
* 状态同步与消息广播
* 断线重连支持
* 可见性裁剪与客户端视图分发

不负责：

* 自己另写一套独立规则
* UI 表现
* Unity 侧动画与界面逻辑

`Server` 必须以 `RuleCore` 为唯一规则来源。

---

### 3.4 UnityClient

Unity 客户端工程。

负责：

* UI 展示
* 玩家输入与交互
* 动画、日志、提示、面板表现
* 网络客户端通信
* 本地视图状态维护
* 必要的本地预演 / 高亮辅助 / 单机调试支持

不负责：

* 最终规则裁定
* 最终合法性判断
* 自行维护独立权威状态

`UnityClient` 可以引用 `RuleCore` 进行本地辅助，但不能取代服务器权威裁定。

---

## 4. 依赖方向

项目依赖方向必须保持单向明确，不得反转：

RuleCore <- Server
RuleCore <- UnityClient

即：

* `Server` 依赖 `RuleCore`
* `UnityClient` 依赖 `RuleCore`
* `RuleCore` 不依赖 `Server`
* `RuleCore` 不依赖 `UnityClient`
* `Server` 与 `UnityClient` 不应形成相互引用

### 4.1 依赖原则说明

#### RuleCore 为底层核心

`RuleCore` 必须保持为纯规则层，不引入 Unity、网络框架或表现层依赖。

#### Server 为权威宿主

`Server` 持有真实对局状态，通过调用 `RuleCore` 完成裁定。

#### UnityClient 为表现与输入端

`UnityClient` 主要负责展示与交互。
即使客户端本地引用 `RuleCore` 做预演，也不能替代服务器的最终裁定。

## 5. 规则来源与文档优先级

本项目中的规则来源，优先级如下：

### 第一优先级：规则书与 FAQ

它们定义游戏正式规则与裁定习惯，是最上位的规则来源。

### 第二优先级：详细设计文档

它们将规则书中的规则进一步整理为结构化设计，是当前开发的直接设计依据。

### 第三优先级：工程设计文档

它们负责回答“这些规则如何在工程里落地”，不能擅自改写上位规则。

### 第四优先级：现有代码实现

若现有代码与规则文档冲突，以文档为准，不以“已有代码能跑”为理由静默修改规则。

## 6. 核心工程原则

### 6.1 服务器权威

凡涉及规则裁定、状态推进、合法性判断的内容，均应以服务器中的权威状态为准。

客户端不应自行决定：

* 某动作是否合法
* 某效果是否成功结算
* 某隐藏信息应否公开
* 某状态是否真正生效

---

### 6.2 规则层与表现层分离

必须严格区分：

* 规则对象
* 视图对象
* 动画对象
* 网络消息对象
* UI 面板状态

不得将核心规则直接散落在：

* UI 按钮回调
* MonoBehaviour 表现脚本
* 临时测试逻辑中

---

### 6.3 显式状态对象优先

项目应优先使用显式状态建模，而不是依赖隐式流程或零散布尔值。

例如：

* `GameState`
* `PlayerState`
* `CardInstance`
* `CharacterInstance`
* `ActionChain`
* `ResponseWindow`
* `InputContext`
* `DamageContext`
* `StatusInstance`

状态对象应具备明确职责，并尽量朝可保存、可恢复的方向设计。

---

### 6.4 Definition / Instance 分离

凡涉及卡牌、角色、异变、状态等内容，应尽量区分：

* Definition：定义层、配置层
* Instance：运行时实例层

例如：

* `CardDefinition` ≠ `CardInstance`
* `CharacterDefinition` ≠ `CharacterInstance`

禁止将“静态定义信息”和“对局中临时状态”混写在同一个对象中。

---

### 6.5 通用骨架优先于个例硬编码

开发时应先实现：

* 数据结构
* 状态对象
* 通用链路
* 通用接口

再考虑：

* 特定角色卡
* 特定宝具
* 特定异变
* 特例牌面文本

不得一开始就将个例硬编码进底层通用模块。

---

### 6.6 中间态可恢复

本项目为联机对战项目，必须考虑：

* 断线重连
* 响应窗口中断
* 输入上下文未完成
* 结算链进行中恢复
* 状态同步重建

因此关键运行时对象应尽量以“可序列化 / 可重建”的思路设计。

## 7. 当前阶段开发顺序

当前阶段的总体顺序为：

### 第一阶段：建立 RuleCore 地基

目标：

* 完成对局状态骨架
* 完成第一批核心类结构
* 建立动作请求与结算链基础
* 建立伤害 / 状态 / 区域等通用规则容器

### 第二阶段：建立 Server 骨架

目标：

* 房间与对局会话
* 请求接收与分发
* 调用 RuleCore 进行裁定
* 状态同步与广播

### 第三阶段：建立 UnityClient 骨架

目标：

* 基础 UI
* 输入与网络接入
* 状态展示
* 日志与提示
* 动画与表现

### 第四阶段：联调与完整规则扩展

目标：

* 完整角色 / 宝具 / 异变接入
* 响应窗口与复杂结算联调
* 可见性裁剪与断线恢复完善
* UI 与表现层补全

---

## 8. 当前阶段暂不处理的内容

在 `RuleCore` 初期阶段，以下内容不是首要目标：

* 完整 UI 美术表现
* 复杂动画系统
* 完整回放系统
* 丰富的客户端体验优化
* 所有卡牌个例的完整落地
* 大规模资源管理与发布流程

初期重点应放在：

* 规则骨架正确
* 对象边界清晰
* 命名稳定
* 状态可推进
* 后续可扩展

## 10. 本文档的使用方式

本文档用于约束整个项目的总体工程方向。

在阅读或编写后续工程设计文档时，应始终以本文档为入口，并遵守以下规则：

* 不得违反本文档定义的工程拆分
* 不得破坏依赖方向
* 不得将 `RuleCore` 污染为 Unity 或 Server 专属实现
* 不得将客户端扩大为权威裁定端
* 若后续某篇工程文档与本文档冲突，应优先修正文档结构，再继续开发

## 11. 总结

《新月花冠》电子版项目的开发方式，不是先做一个“能跑的客户端”，再逐步打补丁式补规则，
而是先建立规则核心，再建立服务器宿主，最后建立客户端表现层。

整体方向如下：

* `RuleCore`：规则真相
* `Server`：权威裁定与状态宿主
* `UnityClient`：输入、展示与表现层

本文档作为工程总览，负责定义这三者的关系、顺序与边界。
后续所有工程设计文档，都应围绕这一总体框架展开。
