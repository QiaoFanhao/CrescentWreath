# 04_41_开发优先级与里程碑

## 1. 文档目的

本文档用于定义《新月花冠》电子版项目当前阶段的开发优先级、阶段目标与里程碑划分。

本文档的目标不是列出所有未来可能实现的功能，也不是做一份精确到日期的项目排期表，而是回答以下问题：

- 当前阶段什么最重要
- 哪些内容必须先做
- 哪些内容可以后做
- 每个阶段做到什么程度，才算进入下一阶段
- 哪些内容当前明确不应抢跑

本文档是 `04_工程设计` 目录中的实施顺序文档。
其作用是为人工开发与 Codex 协作提供统一的“先后顺序”约束，避免项目重新回到“想到什么做什么”的混乱状态。

---

## 2. 项目当前阶段判断

当前项目处于：

> **规则核心从设计转入工程落地的初始阶段**

此阶段的关键任务，不是做出一个外观可玩的 Unity 客户端，也不是尽快接网络联机，而是先把以下事情建立起来：

- RuleCore 的静态骨架
- RuleCore 的基本状态推进能力
- RuleCore 的动作入口与结算链
- RuleCore 的基础规则容器
- 项目整体工程边界

换言之，当前阶段最重要的不是“看起来能玩”，而是：

> **规则世界在代码里能稳定存在。**

---

## 3. 总体开发原则

### 3.1 先 RuleCore，后 Server，再 UnityClient

本项目的开发主线必须遵循以下顺序：

```text
RuleCore -> Server -> UnityClient
```

原因如下：

* `RuleCore` 是规则真相核心
* `Server` 是权威裁定宿主
* `UnityClient` 是输入与表现层

若跳过 `RuleCore` 先做客户端，将极易出现：

* UI 反向定义规则
* 规则散落在 MonoBehaviour 中
* 网络接入后发现状态结构不适配
* 后续大面积返工

---

### 3.2 先骨架，后细节

每一阶段都应优先建立：

* 数据结构
* 状态对象
* 规则容器
* 入口与边界

而不是先实现：

* 个别角色完整技能
* 个别宝具完整特效
* 个别 UI 面板表现
* 大量个例脚本

---

### 3.3 先闭环，后扩张

每一阶段的目标都应优先追求“最小可闭环”，而不是追求“一开始什么都支持”。

例如：

* 先让 ActionRequest 能进入 RuleCore
* 先让 ActionChain 能推进
* 先让 DamageContext 能跑通一条伤害链
* 先让 ResponseWindow 能正式存在

而不是一开始就要求：

* 所有卡牌都能打
* 所有状态都完整支持
* 所有 UI 都已表现完毕

---

### 3.4 以文档为上位约束

开发顺序必须服从：

* 规则书
* FAQ
* 详细设计文档
* 工程设计文档

不得因为某个功能“更容易先做”，就擅自改变工程顺序或规则语义。

---

## 4. 当前阶段的最高优先级

当前最高优先级只应聚焦于以下四件事：

### 4.1 建立 RuleCore 的正式目录与基础项目结构

目标：

* 形成纯 C# 类库工程
* 建立模块目录边界
* 明确命名空间与文件结构

### 4.2 建立 RuleCore 第一批核心类

目标：

* 建立 `GameState`
* 建立 `PlayerState` / `TeamState`
* 建立 `CardInstance` / `CharacterInstance`
* 建立 `ZoneState`
* 建立 `ActionRequest`
* 建立 `ActionChainState`
* 建立 `ResponseWindowState`
* 建立 `InputContextState`
* 建立 `DamageContext`
* 建立 `KillContext`
* 建立 `StatusInstance`
* 建立 `GameEvent`

### 4.3 建立最小规则闭环

目标：

* 能创建一局初始状态
* 能接受一个基本动作请求
* 能开启一条基础动作链
* 能更新一部分正式状态
* 能输出基础规则事件

### 4.4 建立最小验证路径

目标：

* RuleCore 可以通过简单测试或控制台方式验证
* 第一批核心对象不是“只有类型定义”，而是真正可以跑一条最小流程

## 5. 阶段划分

当前建议将项目初期开发划分为以下阶段。

---

## 阶段 A：RuleCore 静态骨架阶段

### 5.1 阶段目标

建立 RuleCore 的第一批静态结构，使游戏的核心规则对象在代码中正式存在。

### 5.2 本阶段应完成的内容

* RuleCore 工程建立
* 目录结构建立
* ID 类型建立
* `GameState` / `PlayerState` / `TeamState`
* `CardInstance` / `CharacterInstance`
* `ZoneKey` / `ZoneState` / `CardMoveReason`
* `ActionRequest` 基类与第一批子类
* `ActionChainState`
* `EffectFrame`
* `ResponseWindowState`
* `InputContextState`
* `DamageContext`
* `KillContext`
* `StatusInstance`
* `GameEvent` 基础结构

### 5.3 本阶段完成判定

满足以下条件时，可认为阶段 A 完成：

* RuleCore 已具备正式目录结构
* 第一批核心类已建立
* 命名体系基本稳定
* 类职责边界基本清晰
* 代码未明显突破 `04_01` 与 `04_02` 边界

### 5.4 本阶段明确不做的内容

* 完整 Definition 系统
* 所有规则细节全部实现
* Unity 场景接入
* 网络接入
* 完整 UI
* 所有卡牌个例实现

---

## 阶段 B：RuleCore 最小动态闭环阶段

### 5.5 阶段目标

让 RuleCore 不仅“有骨架”，而且能跑通最小规则流程。

### 5.6 本阶段应完成的内容

* 创建初始 `GameState`
* 基础 `ActionRequest` 进入系统
* `ActionChainState` 可以推进
* `EffectFrame` 可以被消费
* 可以开启/关闭基础 `ResponseWindowState`
* 可以创建/关闭基础 `InputContextState`
* 可以完成一条最小区域移动流程
* 可以完成一条最小伤害流程
* 可以输出对应基础事件

### 5.7 推荐的最小闭环示例

至少完成以下类型中的一种或数种：

#### 闭环示例 1：基础打牌进入链

* 玩家提交 `PlayCardActionRequest`
* 系统接收请求
* 创建 `ActionChainState`
* 产生至少一个 `EffectFrame`
* 卡牌从手牌移动到指定区域
* 输出移动事件

#### 闭环示例 2：基础伤害链

* 某动作造成一次伤害
* 创建 `DamageContext`
* 计算最终伤害结果
* 更新生命值
* 输出伤害事件与生命变化事件

#### 闭环示例 3：基础响应窗口

* 动作进入结算
* 过程中创建 `ResponseWindowState`
* 等待响应输入
* 关闭窗口并继续推进

### 5.8 本阶段完成判定

满足以下条件时，可认为阶段 B 完成：

* RuleCore 可以从一个请求推进出一个正式结果
* `GameState` 可以在流程中更新
* 事件可以被产出
* 中间态对象不是摆设，能真实参与流程

### 5.9 本阶段明确不做的内容

* 完整联网
* 客户端动画
* 完整卡表接入
* 复杂效果 DSL 完整实现

---

## 阶段 C：RuleCore 核心规则可用阶段

### 5.10 阶段目标

让 RuleCore 具备支持 Server 接入的基础规则能力。

### 5.11 本阶段应优先补强的内容

* 区域移动规则进一步完善
* 伤害 / 防御 / 击杀链补强
* 状态异常统一施加与清除
* 第一批角色技能入口
* 第一批宝具 `Play / Summon / DefensePlace` 语义支持
* 当前异变运行时占位或最小支持
* 基础日志与调试事件完善

### 5.12 本阶段完成判定

满足以下条件时，可认为阶段 C 完成：

* RuleCore 已不是单纯骨架
* 具备支撑 Server 原型接入的最小规则能力
* 主要状态对象与结算链已较稳定
* 后续接网络不需要再推翻核心对象结构

### 5.13 本阶段明确不做的内容

* 所有卡牌全部实现
* 全量 UI
* 回放系统
* 完整客户端体验优化

---

## 阶段 D：Server 原型接入阶段

### 5.14 阶段目标

建立一个最小可工作的服务器骨架，让权威裁定真正运行在 Server 中。

### 5.15 本阶段应完成的内容

* Server 工程建立
* 房间 / 对局会话对象建立
* 玩家连接占位与请求接收入口建立
* 将请求转换为 `ActionRequest`
* Server 调用 RuleCore 推进状态
* 将结果组织为可同步数据
* 形成最小状态同步原型

### 5.16 本阶段完成判定

满足以下条件时，可认为阶段 D 完成：

* Server 可以持有一局正式状态
* Server 可以调用 RuleCore 处理至少一类动作
* Server 可以向外输出处理结果

---

## 阶段 E：UnityClient 原型接入阶段

### 5.17 阶段目标

建立最小 Unity 客户端原型，使其能作为输入与展示端连接 Server。

### 5.18 本阶段应完成的内容

* UnityClient 工程建立
* 基础 UI 与输入结构建立
* 网络接收与发送通路建立
* 状态展示基础框架建立
* 基础日志 / 提示展示建立
* 必要的本地辅助验证（可选）

### 5.19 本阶段完成判定

满足以下条件时，可认为阶段 E 完成：

* UnityClient 可以向 Server 发送一个正式请求
* UnityClient 可以收到状态更新并做基础展示
* Client 侧没有偷偷形成第二套权威规则

## 6. 当前阶段的建议任务顺序

在当前阶段（RuleCore 初期）建议按以下顺序推进任务。

* 建立 ID 类型

### 6.2 第二顺位

* 建立 `ActionRequest`

* 建立 `ResponseWindowState`
* 建立 `InputContextState`

### 6.3 第三顺位

* 建立 `DamageContext`
* 建立 `KillContext`
* 建立 `StatusInstance`
* 建立 `GameEvent`

### 6.4 第四顺位

* 跑通一条最小闭环

## 7. 当前阶段明确不应抢跑的事项

为了避免开发顺序失控，当前阶段明确不应优先投入精力于以下内容：

### 7.1 不抢跑 Unity 表现层

暂不优先做：

* 卡牌动画
* 拖拽手感
* 面板交互细节
* 特效
* 美术资源整理

### 7.2 不抢跑完整 Server 协议

暂不优先做：

* 完整消息协议设计
* 完整断线重连策略
* 完整房间系统
* 复杂同步策略

### 7.3 不抢跑个例卡牌效果

暂不优先做：

* 某一张角色卡的完整脚本
* 某一张宝具的复杂特判
* 某张异变的完整个例流程

### 7.4 不抢跑过度抽象系统

暂不优先做：

* 过于庞大的效果 DSL
* 回放系统
* 完整编辑器工具链
* 为未来十种情况预留过多未验证结构

---

## 8. 每阶段的验收方式

每一阶段结束时，都应至少回答以下问题：

### 8.1 边界是否仍然干净

* RuleCore 是否仍然纯净
* 是否有规则逻辑泄漏到 UI / Unity 脚本
* 是否有客户端偷做权威判断

### 8.2 骨架是否真实可用

* 这些类是否只是空壳
* 是否已有最小流程能跑通
* 是否存在“文档里有、代码里没有”的断裂

### 8.3 命名是否稳定

* 是否出现了不在规范中的别名
* 是否有同一概念多种叫法
* 是否还需要立刻重命名修正

### 8.4 是否具备进入下一阶段的条件

* 当前阶段未完成的问题，是否会阻碍下一阶段
* 若会阻碍，应先补齐而不是硬推进

---

## 9. 与 Codex 协作时的使用方式

在与 Codex 协作开发时，应将本文档作为“任务先后顺序”的上位依据。

### 9.1 默认要求

当用户未特别说明时，Codex 应优先服务当前阶段最高优先级内容，而不是随意跨阶段扩展。

### 9.2 若任务跨越阶段

若某任务明显涉及下一阶段内容，Codex 应先指出：

* 当前项目是否已具备进入下一阶段的条件
* 当前是否更适合先补地基
* 是否需要先完成上一阶段未闭合内容

### 9.3 若任务属于低优先级花哨项

例如：

* UI 表现优化
* 动画设计
* 个例技能脚本

应先判断当前是否会破坏开发顺序。
若会，应优先提示其属于后置事项。

## 10. 近期最小里程碑定义

当前建议明确以下三个近期里程碑。

### 里程碑 M1：RuleCore 骨架成立

达成条件：

* 第一批核心类全部落地
* 命名体系稳定
* 目录结构稳定

### 里程碑 M2：RuleCore 最小闭环跑通

达成条件：

* 至少一条动作请求能够驱动状态变化
* 至少一条伤害或区域移动流程跑通
* 至少一类事件成功产出

### 里程碑 M3：RuleCore 达到可接 Server 原型水平

达成条件：

* 主要状态对象已较稳定
* 动作入口与结算链可基本工作
* 不需要为接 Server 而推翻当前骨架

---

## 11. 总结

当前项目最重要的不是快速做出“看起来像游戏的东西”，而是先建立一套不会在后续增长中崩坏的规则核心。

因此，当前阶段的优先级必须明确为：

1. `RuleCore` 静态骨架
2. `RuleCore` 最小动态闭环
3. `RuleCore` 核心规则可用
4. `Server` 原型接入
5. `UnityClient` 原型接入

本项目的开发节奏，应始终遵循：

> **先立规则骨架，再接权威宿主，最后接表现层。**

只要这个顺序不乱，项目即使走得慢一些，也仍然是在向真正可完成的方向前进。
若这个顺序被打乱，即使短期功能看起来更多，长期也极易陷入返工与结构崩坏。
